# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Project Service Automation से Project Operations में अपग्रेड करें

हम Microsoft Dynamics 365 Project Service Automation से Microsoft Dynamics 365 Project Operations में अपग्रेड के लिए तीन चरणों में से दूसरे की घोषणा करते हुए उत्साहित हैं. यह लेख उन ग्राहकों के लिए एक सिंहावलोकन प्रदान करता है जो इस रोमांचक यात्रा को शुरू कर रहे हैं। 

उन्नयन वितरण कार्यक्रम को तीन चरणों में विभाजित किया जाएगा।

| डिलीवरी अपग्रेड करें | चरण 1 (जनवरी 2022) | चरण 2 (नवंबर 2022) | चरण 3 (अप्रैल वेव 2023)  |
|------------------|------------------------|---------------------------|---------------------------|
| परियोजनाओं के लिए वर्क ब्रेकडाउन स्ट्रक्चर (WBS) पर कोई निर्भरता नहीं | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| Project Operations की वर्तमान में समर्थित सीमाओं के भीतर एक WBS | | :heavy_check_mark: | :heavy_check_mark: |
| Project Desktop Client के लिए समर्थन सहित, Project Operations की वर्तमान में समर्थित सीमाओं के बाहर एक WBS | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>प्रक्रिया फीचर अपग्रेड करें 

नवीनीकरण प्रक्रिया के भाग के रूप में, हमने व्यवस्थापकों को विफलताओं का अधिक आसानी से निदान करने में सक्षम बनाने के लिए साइट मानचित्र में अपग्रेड लॉग जोड़े हैं। नए इंटरफ़ेस के अलावा, अपग्रेड के बाद डेटा अखंडता सुनिश्चित करने के लिए नए सत्यापन नियम जोड़े जाएंगे। निम्नलिखित सत्यापन अपग्रेड प्रक्रिया में जोड़े जाएंगे।

| सत्यापन | चरण 1 (जनवरी 2022) | चरण 2 (नवंबर 2022) | फेज़ 3  |
|-------------|------------------------|---------------------------|---------------------------|
| WBS को सामान्य डेटा अखंडता उल्लंघनों के खिलाफ मान्य किया जाएगा (उदाहरण के लिए, संसाधन असाइनमेंट जो एक ही मूल कार्य से जुड़े हैं लेकिन अलग-अलग मूल प्रोजेक्ट हैं)। | | :heavy_check_mark: | :heavy_check_mark: |
| [Project for the Web की ज्ञात सीमाएँ।](/project-for-the-web/project-for-the-web-limits-and-boundaries) WBS के विरुद्ध मान्य किया जाएगा। | | :heavy_check_mark: | :heavy_check_mark: |
| Project desktop client की ज्ञात सीमाएँ WBS के विरुद्ध मान्य की जाएगी। | |  | :heavy_check_mark: |
| सामान्य असंगत कैलेंडर नियम अपवादों के विरुद्ध बुक करने योग्य संसाधनों और प्रोजेक्ट कैलेंडर का मूल्यांकन किया जाएगा। | | :heavy_check_mark: | :heavy_check_mark: |

चरण 2 में, जो ग्राहक Project Operations में अपग्रेड करते हैं, उनके मौजूदा प्रोजेक्ट प्रोजेक्ट प्लानिंग के लिए केवल-पढ़ने के अनुभव के लिए अपग्रेड किए जाएंगे। इस केवल-पढ़ने के अनुभव में, संपूर्ण WBS ट्रैकिंग ग्रिड में दिखाई देगा। WBS को संपादित करने के लिए, परियोजना के मुख्य पृष्ठ पर [**बदलें**](/PSA-Upgrade-Project-Conversion.md) परियोजना प्रबंधक चुन सकते हैं। एक पृष्ठभूमि प्रक्रिया तब प्रोजेक्ट को अपडेट करती है ताकि यह Project for the Web से नए प्रोजेक्ट शेड्यूलिंग अनुभव का समर्थन करे। यह चरण उन ग्राहकों के लिए उपयुक्त है जिनके पास ऐसी परियोजनाएं हैं जो [Project for the Web की ज्ञात सीमाएँ](/project-for-the-web/project-for-the-web-limits-and-boundaries) के भीतर फिट होती हैं।

चरण 3 में, Project Desktop Client के लिए समर्थन जोड़ा जाएगा, उन ग्राहकों के लाभ के लिए जो उस एप्लिकेशन से अपनी परियोजनाओं को संपादित करना जारी रखना चाहते हैं। हालांकि, यदि मौजूदा प्रोजेक्ट्स को वेब अनुभव के लिए नए प्रोजेक्ट में कनवर्ट किया जाता है, तो प्रत्येक कनवर्ट किए गए प्रोजेक्ट के लिए ऐड-इन तक पहुंच अक्षम कर दी जाएगी।

## <a name="prerequisites"></a>पूर्वावश्यकताएँ

चरण 1 के उन्नयन के लिए पात्र होने के लिए, आपको निम्नलिखित मानदंडों को पूरा करना होगा:

- लक्ष्य परिवेश को **msdyn_projecttask** कंपनी में कोई रिकॉर्ड नहीं होना चाहिए।
- वैध Project Operations लाइसेंस सभी सक्रिय उपयोगकर्ताओं को सौंपे जाने चाहिए। 
- आपको अपग्रेड प्रक्रिया को कम से कम एक गैर-उत्पादन परिवेश में सत्यापित करना होगा जिसमें एक प्रतिनिधि डेटासेट हो जो आपके उत्पादन परिवेश के साथ संरेखित हो।
- लक्ष्य परिवेश को Project Service Automation Update रिलीज़ 37 (V3.10.58.120) या बाद के संस्करण में अद्यतन किया जाना चाहिए।

चरण 2 के उन्नयन के लिए पात्र होने के लिए, आपको निम्नलिखित मानदंडों को पूरा करना होगा:

- वैध Project Operations लाइसेंस सभी सक्रिय उपयोगकर्ताओं को सौंपे जाने चाहिए। 
- आपको अपग्रेड प्रक्रिया को कम से कम एक गैर-उत्पादन परिवेश में सत्यापित करना होगा जिसमें एक प्रतिनिधि डेटासेट हो जो आपके उत्पादन परिवेश के साथ संरेखित हो।
- लक्ष्य परिवेश को Project Service Automation Update रिलीज़ 37 (V3.10.58.120) या बाद के संस्करण में अद्यतन किया जाना चाहिए।
- ऐसे वातावरण जिनमें कार्य होते हैं (msdyn_projecttask) केवल तभी समर्थित होते हैं जब प्रति प्रोजेक्ट कार्यों की कुल संख्या 500 या उससे कम हो।

चरण 3 के लिए पूर्वापेक्षाएँ सामान्य उपलब्धता तिथि के निकट आने पर अद्यतन की जाएंगी।

## <a name="licensing"></a>लाइसेंसिंग

यदि आपके पास Project Service Automation के लिए सक्रिय लाइसेंस हैं, तो आप Project Operations को स्थापित और उपयोग कर सकते हैं, जिसमें Project Service Automation की सभी क्षमताएँ और बहुत कुछ शामिल हैं। इस तरह, जब आप उत्पादन में Project Service Automation का उपयोग करना जारी रखते हैं, तब आप Project Operations की क्षमताओं का परीक्षण कर सकते हैं। आपके Project Service Automation लाइसेंस की समय सीमा समाप्त होने के बाद, आपको Project Operations में संक्रमण करना होगा। जब आप इस संक्रमण की योजना बनाते हैं, तो आपको इस तथ्य पर ध्यान देना चाहिए कि Project Operations लाइसेंस में Project Service Automation लाइसेंस शामिल नहीं है। जिन ग्राहकों के पास ऐसे परिदृश्य हैं जहां उन्होंने Project Service Automation को परिनियोजित किया है और Project Operations में जाने की योजना बनाते समय PSA के लिए अपने लाइसेंस का उपयोग जारी रखने या बढ़ाने की आवश्यकता है, वे खरीदे गए Project Operations लाइसेंस के आधार पर अस्थायी PSA लाइसेंस का अनुरोध कर सकते हैं। एक Project Operations लाइसेंस के लिए एक Project Service Automation लाइसेंस जारी किया जाएगा। इस लिंक का उपयोग करके अस्थायी पीएसए लाइसेंस का अनुरोध किया जा सकता है: aka.ms/ineedpsa

## <a name="testing-and-refactoring-customizations"></a>अनुकूलन का परीक्षण और रीफैक्टरिंग

एक प्रारंभिक बिंदु के रूप में, सभी अनुकूलन को एक स्वच्छ Project Operations (लाइट) वातावरण में आयात करें ताकि यह पुष्टि हो सके कि आयात सफल है, और व्यवसाय संचालन अपेक्षा के अनुरूप व्यवहार करता है।

यहां देखने के लिए कुछ चीजें दी गई हैं:

- अनुपलब्ध निर्भरता के कारण आयात विफल हो सकता है। दूसरे शब्दों में, अनुकूलन संदर्भ फ़ील्ड या अन्य घटक जिन्हें Project Operations में हटा दिया गया है। इस स्थिति में, इन निर्भरताओं को विकास परिवेश से हटा दें।
- यदि आपके अप्रबंधित और प्रबंधित समाधानों में ऐसे घटक शामिल हैं जो अनुकूलित नहीं हैं, तो उन घटकों को समाधान से निकाल दें। उदाहरण के लिए, जब आप **परियोजना** निकाय को अनुकूलित करते हैं, अपने समाधान में केवल निकाय शीर्षलेख जोड़ें। सभी फ़ील्ड न जोड़ें। यदि आपने पहले सभी उप-घटकों को जोड़ा है, तो आपको मैन्युअल रूप से एक नया समाधान बनाना होगा और उसमें प्रासंगिक घटक जोड़ना होगा।
- प्रपत्र और दृश्य अपेक्षित रूप से प्रकट नहीं हो सकते हैं। कुछ परिस्थितियों में, यदि आपने किसी आउट-ऑफ़-बॉक्स प्रपत्र या दृश्य को अनुकूलित किया है, तो अनुकूलन Project Operations में नए अपडेट को प्रभावी होने से रोक सकता है। इन मुद्दों की पहचान करने के लिए, हम अनुशंसा करते हैं कि आप Project Operations की क्लीन इंस्टॉलेशन और Project Operations की स्थापना की साथ-साथ समीक्षा करें जिसमें आपके अनुकूलन शामिल हैं। यह पुष्टि करने के लिए अपने व्यवसाय में सबसे अधिक उपयोग किए जाने वाले फ़ॉर्म की तुलना करें कि फ़ॉर्म का आपका संस्करण अभी भी समझ में आता है और फ़ॉर्म के स्वच्छ संस्करण से कुछ छूट नहीं रहा है। आपके द्वारा अनुकूलित किए गए किसी भी दृश्य के लिए उसी प्रकार की साथ-साथ समीक्षा करें।
- व्यावसायिक तर्क रनटाइम पर विफल हो सकता है। चूंकि आयात के समय आपके प्लग-इन में फ़ील्ड के संदर्भ मान्य नहीं हैं, इसलिए उन फ़ील्ड के संदर्भों के कारण व्यावसायिक तर्क विफल हो सकता है जो अब मौजूद नहीं हैं, और आपको निम्न उदाहरण के जैसा एक त्रुटि संदेश प्राप्त हो सकता है: "'प्रोजेक्ट' इकाई में नाम = 'msdyn_plannedhours' और NameMapping = 'लॉजिकल' के साथ विशेषता नहीं है।" इस मामले में, अपने अनुकूलन संशोधित करें ताकि वे नए क्षेत्रों का उपयोग करें। यदि आप अपने प्लग-इन लॉजिक में स्वचालित रूप से जेनरेट की गई प्रॉक्सी कक्षाओं और मजबूत प्रकार के संदर्भों का उपयोग करते हैं, तो उन प्रॉक्सी को क्लीन इंस्टॉलेशन से पुन: उत्पन्न करने पर विचार करें। इस तरह, आप आसानी से उन सभी स्थानों की पहचान कर सकते हैं जहां आपके प्लग-इन बहिष्कृत फ़ील्ड पर निर्भर करते हैं।

Project Operations को साफ़-साफ़ आयात करने के लिए अपने अनुकूलनों को अद्यतन करने के बाद, अगले चरणों पर जाएँ।

## <a name="end-to-end-testing-in-development-environments"></a>विकास के वातावरण में एंड-टू-एंड परीक्षण

### <a name="initiate-upgrade"></a>अपग्रेड शुरू करें 

1. Power Platform व्यवस्थापक केंद्र में, अपना परिवेश ढूंढें और चुनें. फिर, अनुप्रयोगों में, **Dynamics 365 Project Operations** ढूंढें और चुनें.
2. अपग्रेड शुरू करने के लिए **स्थापित करें** चुनें. Power Platform व्यवस्थापक केंद्र इस स्थापना को एक नई स्थापना के रूप में प्रस्तुत करेगा। हालांकि, Project Service Automation के पुराने संस्करण की उपस्थिति का पता लगाया जाएगा, और मौजूदा स्थापना को अपग्रेड किया जाएगा।

    नवीनीकरण पूर्ण होने के बाद, परिवेश को यह दिखाना चाहिए कि Project Operations स्थापित है, और Project Service Automation स्थापित नहीं है।

    परिवेश में डेटा की मात्रा के आधार पर, अपग्रेड को पूरा करने में कई घंटे लग सकते हैं. अपग्रेड का प्रबंधन करने वाली कोर टीम को तदनुसार योजना बनानी चाहिए और गैर-व्यावसायिक घंटों के दौरान अपग्रेड चलाना चाहिए। कुछ मामलों में, यदि डेटा वॉल्यूम बड़ा है, तो नवीनीकरण सप्ताहांत के दौरान चलाया जाना चाहिए। शेड्यूलिंग के बारे में निर्णय निचले वातावरण में परीक्षण के परिणामों पर आधारित होना चाहिए।

3. उपयुक्त के रूप में कस्टम समाधान अपग्रेड करें। इस बिंदु पर, इस आलेख [अनुकूलन का परीक्षण और रीफैक्टरिंग](#testing-and-refactoring-customizations) खंड में के आपके द्वारा अपने अनुकूलन में किए गए किसी भी परिवर्तन को लागू करें।
4. **make.powerapps.com** पर जाएँ, पोर्टल के शीर्ष दाईं ओर स्थित ड्रॉप डाउन से अपना परिवेश चुनें, बाएं मेनू से **समाधान** चुनें, **Project Operations बहिष्कृत घटक** समाधान चुनें और **स्थापना रद्द करें**.

    यह समाधान एक अस्थायी समाधान है जो मौजूदा डेटा मॉडल और अपग्रेड के दौरान मौजूद घटकों को रखता है। इस समाधान को हटाकर, आप उन सभी फ़ील्ड और घटकों को हटा देते हैं जिनका अब उपयोग नहीं किया जाता है। इस तरह, आप इंटरफ़ेस को सरल बनाने और एकीकरण और विस्तार को आसान बनाने में मदद करते हैं।
    
### <a name="upgrade-to-project-operations-lite"></a>Project Operations लाइट में अपग्रेड करें

निम्न चरण नवीनीकरण प्रक्रिया और संबद्ध त्रुटि लॉगिंग का वर्णन करते हैं:

1. **पीएसए संस्करण की जाँच करें:** Project Operations को स्थापित करने के लिए, आपके पास V3.10.58.120 या उच्चतर होना चाहिए।
1. **पूर्व-सत्यापन:** जब कोई व्यवस्थापक नवीनीकरण आरंभ करता है, तो सिस्टम प्रत्येक निकाय पर एक पूर्व-सत्यापन कार्रवाई चलाता है जो Project Operations समाधान का मूल है। यह चरण सत्यापित करता है कि सभी निकाय संदर्भ मान्य हैं, और यह सुनिश्चित करता है कि WBS से संबंधित डेटा Project for the Web की प्रकाशित सीमाओं के भीतर है।
1. **मेटाडेटा अपग्रेड:** सफल पूर्व-सत्यापन के बाद, सिस्टम स्कीमा में परिवर्तन शुरू करता है और एक बहिष्कृत घटक समाधान बनाता है। अनुकूलन की सभी आवश्यक रीफैक्टरिंग पूरी करने के बाद आप इस बहिष्कृत समाधान को निकाल सकते हैं। यह चरण अपग्रेड प्रक्रिया का सबसे लंबा हिस्सा है और इसे पूरा होने में चार घंटे तक लग सकते हैं।
1. **डेटा अपग्रेड:** मेटाडेटा अपग्रेड चरण में सभी आवश्यक स्कीमा परिवर्तन पूर्ण होने के बाद, आपका डेटा नए स्कीमा में माइग्रेट कर दिया जाता है, और कोई भी आवश्यक डिफ़ॉल्ट और पुनर्गणना की जाती है।
1. **प्रोजेक्ट शेड्यूल इंजन अपडेट:** सफल डेटा अपग्रेड के बाद, मुख्य पृष्ठ पर **अनुसूची** टैब को **कार्य** पुनः लेबल किया गया है. जब कोई उपयोगकर्ता अपग्रेड के बाद इस टैब का चयन करता है, तो उन्हें WBS का केवल-पढ़ने के लिए संस्करण देखने के लिए ट्रैकिंग ग्रिड पर नेविगेट करने के लिए निर्देशित किया जाता है। WBS को संपादित करने के लिए, उन्हें [रूपांतरण प्रक्रिया](/PSA-Upgrade-Project-Conversion.md) शेड्यूल आरंभ करना होगा. पूर्व-मौजूदा WBS के बिना सभी परियोजनाएं रूपांतरण के बिना सीधे नए शेड्यूलिंग अनुभव का उपयोग कर सकती हैं।
 
### <a name="validate-common-scenarios"></a>सामान्य परिदृश्य सत्यापित करें

जब आप अपने विशिष्ट अनुकूलनों को मान्य करते हैं, तो हम अनुशंसा करते हैं कि आप उन व्यावसायिक प्रक्रियाओं की भी समीक्षा करें जो सभी अनुप्रयोगों में समर्थित हैं। इन व्यावसायिक प्रक्रियाओं में शामिल हैं, लेकिन इन तक सीमित नहीं हैं, बिक्री संस्थाओं जैसे कि कोटेशन और अनुबंधों का निर्माण, और परियोजनाओं का निर्माण जिसमें WBS और वास्तविक का अनुमोदन शामिल है।

## <a name="major-changes-between-project-service-automation-and-project-operations"></a>Project Service Automation से Project Operations के बीच मुख्य बदलाव

यह खंड उन प्रमुख परिवर्तनों का सारांश प्रदान करता है जिनकी आप Project Service Automation और Project Operations के बीच अपेक्षा कर सकते हैं।

### <a name="project-planning"></a>परियोजना नियोजन

Project Operations में प्रोजेक्ट प्लानिंग क्षमताएं अब क्लाइंट-साइड लॉजिक और सर्वर-साइड लॉजिक के संयोजन पर निर्भर नहीं हैं। बजाय, Project Operations प्राथमिक शेड्यूलिंग इंजन के रूप में Project for the Web का उपयोग करता है। शेड्यूलिंग क्षमताओं में यह परिवर्तन कई नई सुविधाओं को सक्षम बनाता है, जैसे बोर्ड और गैंट दृश्य, संसाधन-संचालित योजना, [कार्य चेकलिस्ट आइटम](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c), और प्रोजेक्ट शेड्यूलिंग मोड। नई शेड्यूलिंग क्षमताएं भी नए [एप्लिकेशन प्रोग्रामिंग इंटरफेस (एपीआई)](../project-management/schedule-api-preview.md) के समृद्ध सेट द्वारा समर्थित हैं। इन एपीआई का उद्देश्य यह सुनिश्चित करने में मदद करना है कि डब्ल्यूबीएस में किसी इकाई को बनाने, अपडेट करने या हटाने के लिए कोई प्रोग्रामेटिक ऑपरेशन शेड्यूल में परिकलित फ़ील्ड को दूषित नहीं करता है।

### <a name="billing-and-pricing"></a>बिलिंग और मूल्य निर्धारण

Project Operations में निरंतर निवेश के हिस्से के रूप में, बिलिंग और मूल्य निर्धारण में कई नई क्षमताएं उपलब्ध हैं। यहाँ कुछ उदाहरण दिए गए हैं:

- [उप-अनुबंध प्रबंधन](../pro/subcontracting/managing-subcontracts-overview.md)
- [एडवांसेस और रिटेनर-आधारित अनुबंध](../pro/sales/set-up-advances-retainer-based-contracts-sales.md)
- [सीमा-में-रखें की स्थिति और मान्यताएँ अनुबंध](../pro/proforma-invoicing/manage-nte-status-validations-sales.md)
- कार्य-आधारित बिलिंग

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>अपग्रेड टूलिंग उपलब्ध होने से पहले मैं Project Operations को कैसे स्थापित कर सकता हूं?

अपग्रेड टूलिंग उपलब्ध होने से पहले Project Operations को स्थापित करने के लिए दो विकल्प हैं:

- नए परिवेश का प्रावधान।
- किसी भी बिक्री संगठन पर Project Operations को अलग से परिनियोजित करें जहां Project Service Automation मौजूद नहीं है।

यदि किसी संगठन पर Project Service Automation स्थापित है, लेकिन उसका उपयोग नहीं किया गया था, तो उसे अनइंस्टॉल किया जा सकता है। आपके द्वारा Project Service Automation को पूरी तरह से हटाने के बाद, Project Operations को उसी संगठन पर स्थापित किया जा सकता है।
